# 红薯百宝箱去水印生成文案项目架构设计

## 1. 架构目标

本项目为微信小程序产品，核心功能包括小红书去水印、热门标题生成、热门文案生成。

一期目标是快速上线免费工具 MVP，优先保证核心功能闭环、接口安全、后续可扩展，不在一期实现会员、充值、每日次数、分享加次数、广告加次数等复杂增长能力。

架构设计目标：

- 前后端分离，小程序负责页面和交互，后端负责业务逻辑和第三方接口。
- 第三方接口密钥、AI Key、去水印解析接口统一放在后端，避免暴露在小程序端。
- 后端采用模块化设计，方便后续增加次数系统、广告奖励、公众号引导、模板库等功能。
- 一期保持简单，避免过度设计。

## 2. 技术栈

### 2.1 前端

前端使用微信小程序原生语言开发。

主要技术：

- WXML
- WXSS
- JavaScript
- 微信小程序原生 API
- 微信开发者工具

前端职责：

- 页面展示
- 用户输入
- 调用后端接口
- 展示解析和生成结果
- 本地登录态管理
- 基础交互反馈

### 2.2 后端

后端使用 NestJS 开发 REST API 服务。

主要技术：

- NestJS
- TypeScript
- MySQL
- Prisma 或 TypeORM
- JWT
- class-validator
- Swagger

推荐 ORM：

- 一期推荐使用 Prisma，开发效率高，类型提示清晰，适合快速搭建业务 CRUD。

### 2.3 缓存与限流

一期可以不强依赖 Redis。

二期如果增加每日次数、分享加次数、广告加次数、接口限流等能力，建议引入 Redis。

Redis 可用于：

- 接口频率限制
- 每日使用次数统计
- 分享奖励临时状态
- 短期缓存
- 防刷控制

### 2.4 第三方服务

项目可能接入：

- 微信登录接口
- 小红书解析接口
- AI 大模型接口
- 对象存储服务
- 后期微信广告能力

## 3. 整体架构

```text
微信小程序前端
  |
  | HTTPS API
  v
NestJS 后端服务
  |
  |-- MySQL：用户、记录、收藏、配置、日志
  |
  |-- Redis：限流、次数、缓存，二期引入
  |
  |-- 微信接口：登录、用户身份
  |
  |-- 小红书解析服务：去水印解析
  |
  |-- AI 服务：标题生成、文案生成
```

## 5. 前端架构设计

### 5.1 页面结构

一期小程序页面：

- 首页
- 去水印页面
- 去水印结果页面
- 标题生成页面
- 文案生成页面
- 历史记录页面
- 收藏页面
- 我的页面
- 使用说明页面
- 免责声明页面

### 5.2 前端分层

前端建议分为四层：

```text
pages 页面层
  |
components 组件层
  |
services 接口层
  |
utils 工具层
```

各层职责：

- pages：负责页面状态、用户交互、页面跳转。
- components：负责可复用 UI，如功能卡片、结果卡片、空状态、加载状态。
- services：统一封装后端接口调用。
- utils：封装本地缓存、表单校验、格式化等通用逻辑。

### 5.3 请求封装

小程序所有接口请求统一经过 `services/request.js`。

请求封装需要处理：

- baseURL
- token 注入
- loading 状态
- 错误提示
- 登录失效处理
- 统一响应格式解析

建议响应格式：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

### 5.4 登录态管理

前端通过 `wx.login` 获取临时 code，再调用后端登录接口。

登录流程：

```text
小程序 wx.login
  |
  v
POST /api/v1/auth/wechat-login
  |
  v
后端换取 openid
  |
  v
后端创建或更新用户
  |
  v
返回 accessToken 和用户信息
  |
  v
小程序本地保存 token
```

## 6. 后端架构设计

### 6.1 NestJS 分层

后端采用 NestJS 模块化分层：

```text
Controller：接收请求、参数校验、返回响应
Service：业务逻辑
Repository/Prisma：数据库访问
Integration：第三方服务封装
Common：全局过滤器、守卫、拦截器、工具
```

### 6.2 核心模块

#### 6.2.1 AuthModule

职责：

- 微信登录
- 获取 openid
- 创建或更新用户
- 签发 JWT
- 校验登录态

主要接口：


- 查询当前用户信息

- 校验小红书链接
- 调用小红书解析服务
- 保存解析记录
- 返回无水印素材结果

主要接口：

```text

GET /api/v1/watermark/records/:id
```

#### 6.2.4 GenerateModule

职责：

- 生成热门标题

```text
POST /api/v1/generate/titles
```

## 7. 后端通用能力

### 7.1 全局前缀和版本

后端接口统一使用：

```text
/api/v1
```

示例：

```text
POST /api/v1/watermark/parse
```

### 7.2 参数校验

所有用户输入都需要通过 DTO 校验。

示例：

- 链接不能为空
- 链接长度限制
- 文案主题不能为空
- 文案主题长度限制
- 枚举字段只能传指定值

### 7.3 登录鉴权

除部分公开接口外，业务接口默认需要登录。

建议使用：

- JWT
- AuthGuard
- CurrentUser 自定义装饰器

### 7.4 统一异常处理

使用全局异常过滤器统一返回错误结构。

错误响应示例：

```json
{
  "code": 40001,
  "message": "链接格式不正确",
  "data": null
}
```

### 7.5 统一响应处理

成功响应示例：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "id": "record_id"
  }
}
```

### 7.6 日志记录

需要记录：

- 请求路径
- 用户 ID
- 接口耗时
- 第三方接口调用结果
- 异常信息

重点记录：

- 去水印解析失败原因
- AI 生成失败原因
- 微信登录失败原因

## 8. 第三方服务封装

### 8.1 WechatIntegration

职责：

- 调用微信 `jscode2session`
- 根据 code 获取 openid
- 处理微信接口异常

### 8.2 XhsParserIntegration

职责：

- 调用小红书解析服务
- 将第三方响应转换为系统统一格式
- 隐藏第三方接口细节
- 处理解析失败、超时、接口不可用等情况

统一返回结构建议：

```json
{
  "title": "笔记标题",
  "cover": "封面地址",
  "contentType": "image",
  "mediaList": [],
  "content": "原文案"
}
```

### 8.3 AiProviderIntegration

职责：

- 调用 AI 大模型接口
- 管理标题生成提示词
- 管理文案生成提示词
- 过滤和格式化 AI 输出
- 处理超时和生成失败

AI 能力不要散落在业务模块中，应统一由 `ai-provider` 封装。

## 12. 安全设计

### 12.1 小程序端

- 不保存第三方接口密钥。
- 不直接调用 AI 服务。
- 不直接调用小红书解析服务。
- token 存储在本地缓存中。
- 登录失效后重新调用微信登录。

### 12.2 后端

- 所有业务接口默认需要 JWT 鉴权。
- 使用 DTO 校验用户输入。
- 对链接、主题、卖点、文案输入做长度限制。
- AI Key、解析接口 Key、微信 AppSecret 全部通过环境变量配置。
- 记录第三方接口调用日志。
- 二期加入 Redis 限流，防止接口被刷。

## 16. 架构结论

本项目建议采用前后端分离的轻量单体架构。

微信小程序只负责用户界面、交互和接口调用；NestJS 后端负责登录、业务逻辑、数据库、第三方接口和安全控制。

一期保持简单，先跑通免费工具闭环；二期在现有模块基础上增加次数系统、分享奖励、广告奖励、公众号引导和模板库，不需要推倒重来。
